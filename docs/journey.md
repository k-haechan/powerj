# 고객 여정지도 (Customer Journey Map)

> 이 문서는 [problem.md](./problem.md)에서 정의한 제품 개념을 **실제 사용자의 하루**에 대입해, 페르소나별로 어디서 가치를 느끼고 어디서 이탈하는지를 지도로 그린다.

## 제품 한 줄 요약

하루를 **프로세스 체인**으로 설계하고(일부 목표는 당일 **즉석 수립**), 각 프로세스에 **정량 목표**를 걸고, 실행은 **타이머와 방해금지 모드**로 시간까지 재며, 프로세스 전환마다 **달성률·실패 원인 정리를 강제**하고, 하루가 끝나면 **일기와 대시보드로 마감**해 다음날 계획에 되먹이는 극한 계획 앱.

## 페르소나 인덱스

| # | 페르소나 | 한 줄 특징 | 대표 목표 | 핵심 페인 |
|---|---|---|---|---|
| P1 | **자기계발형 직장인/취준생** | 자투리 1분까지 성장에 쓰려는 고밀도 자기관리형 | 이동 중 개발 영상 2개, 책 200pg | 계획은 세우지만 자투리 시간이 새고, 실패를 그냥 넘겨 반복됨 |
| P2 | **수험생/공시생** | 매일 비슷한 시간표, 극단적 시간 규율 | 하루 기출 200문제, 인강 4강 | 공부 "시간"은 재도 "얼마나 해냈는지"는 측정 못 함 |
| P3 | **N잡러/사이드프로젝트 직장인** | 본업 + 사이드, 새벽·출퇴근·저녁을 쥐어짬 | 새벽 1시간 사이드 커밋 1개, 퇴근길 강의 1개 | 파편화된 자투리를 목표화하지 못해 사이드가 흐지부지됨 |
| P4 | **프리랜서/1인 창업가** | 외부 구조가 없어 스스로 하루를 세워야 함 | 오전 딥워크 3시간 산출물 1건, 미팅 후 정리 | 자율성은 높지만 무너지기 쉬워 강제 장치가 필요 |

## 공통 여정 골격 (Journey Backbone)

모든 페르소나는 제품의 하루 루프를 따라 아래 6단계를 지난다. 페르소나별 섹션의 여정 표는 이 골격을 공유하되 내용을 달리한다.

| 단계 | 이름 | 제품에서 하는 일 |
|---|---|---|
| ① | **전날 밤 계획** | 다음날을 프로세스 체인으로 설계, 각 프로세스에 정량 목표 부여 |
| ② | **프로세스 진입** | 사전 목표를 확인하거나, 즉석 구간이면 지금 목표를 수립(시간 기준/목표 기준). 시작과 함께 **타이머·방해금지 모드 ON** |
| ③ | **프로세스 실행** | 목표 수치를 채워가며 실행. 타이머가 실제 소요 시간을 잼 |
| ④ | **전환 검증** | 프로세스 종료 시 달성률(%)·실패 원인 입력 (강제 관문). 타이머 정지 → 시간 효율 확정 |
| ⑤ | **하루 마감** | 일기 작성 + 대시보드로 달성률·**시간 효율(단축/초과)** 확인 |
| ⑥ | **축적·개선** | 주/월 단위로 패턴 확인 → 목표 설정·하루 설계를 현실화 |

## P1 — 자기계발형 직장인/취준생

### 프로필 카드
- **소개**: 낮에는 일(혹은 취업 준비), 그 외 모든 자투리를 성장에 투자하려는 27세. 문서의 원형 사용자.
- **상황/제약**: 출퇴근 왕복 2시간, 점심·저녁 자투리. 시간은 있지만 파편적이고 쉽게 샌다.
- **목표**: "노는 시간 없이" 하루를 성장으로 채우기. 개발 실력·독서량을 정량적으로 늘리기.
- **페인포인트**: 계획은 잘 세우지만 이동 중 유튜브·SNS로 새고, 목표 미달을 그냥 넘겨 같은 실패가 반복된다.
- **대표 정량 목표**: 이동(08:00~09:00) 개발 영상 2개 + 책 200pg까지 / 점심 CS 문제 5개 / 퇴근길 영어 회화 20분.

### 시나리오
> 전날 밤, 내일을 8개 프로세스로 쪼갠다. 출근 이동 08:00~09:00에 "개발 영상 2개, 책 200pg"를 건다. 아침 지하철에서 앱이 이 목표를 띄워주고, 시작을 누르자 **타이머가 돌고 방해금지 모드가 켜진다**. 영상 1개를 보다 졸았다. 9시 회사 도착, 앱이 다음 프로세스(업무)로 넘어가기 전 전환 검증을 막아선다. **달성률 60%, 원인: '수면 부족으로 졸음'** 을 태그한다. 점심에는 예상보다 30분이 비었다 — 이 자투리는 **즉석·시간 기준**으로 "30분에 CS 문제 5개"를 그 자리에서 세워 실행하고, 22분 만에 끝내 **8분 단축**한다. 저녁, 하루를 마감하며 일기를 쓰고 대시보드를 본다. 이동 프로세스 달성률이 이번 주 내내 낮고 실패 원인 1위가 '졸음'인 반면, CS 문제는 늘 목표 시간보다 빨리 끝난다. "이동엔 영상 1개가 적정, CS는 목표를 더 높여도 됨"을 데이터로 깨닫는다.

### 여정 단계
| 단계 | 사용자 행동 | 생각·감정 | 터치포인트 | 페인 & 기회 |
|---|---|---|---|---|
| ① 전날 밤 계획 | 하루를 프로세스로 쪼개고 자투리마다 목표 부여 | "내일은 진짜 꽉 채운다" 기대·의욕 | 계획 편집 화면 | 목표를 과하게 잡는 경향 → 과거 달성률 힌트 제공 |
| ② 프로세스 진입 | 지하철에서 이동 프로세스 목표 확인 | "영상 2개, 책 200pg 가자" | 진행 중 프로세스 카드 | 알림이 늦으면 자투리 초반을 놓침 |
| ③ 프로세스 실행 | 영상 보다 졸음, 1개만 봄 | "아 또 졸았네…" 죄책감 | 목표 진행 위젯 | 진행률 실시간 표시로 각성 유도 |
| ④ 전환 검증 | 60% + '졸음' 태그 입력 | "그냥 넘어가고 싶다" 저항 | 전환 검증 모달 | **강제라서 오히려 기록됨** — 마찰 최소 UX가 관건 |
| ⑤ 하루 마감 | 일기 + 대시보드 확인 | "이동은 늘 낮네" 자각 | 마감/대시보드 화면 | 실패 원인 집계가 통찰을 줌 |
| ⑥ 축적·개선 | 목표를 영상 1개로 하향 | "이제 현실적이다" 안도 | 주간 리포트 | 목표 자동 보정 제안 기회 |

### 핵심 순간 (Key Moments)
- **전환 검증의 저항 → 기록**: "그냥 넘기고 싶은" 순간을 강제로 붙잡는 것이 이 사용자에게 가장 큰 가치이자 가장 큰 이탈 위험. 입력 마찰이 3초를 넘으면 반감한다.
- **대시보드의 자각**: 흩어진 실패가 '졸음 5회'로 집계되는 순간, 처음으로 자기 패턴을 데이터로 마주한다.

## P2 — 수험생/공시생

### 프로필 카드
- **소개**: 하루 10시간 이상 공부하는 24세 공시생. 매일 시간표가 거의 동일.
- **상황/제약**: 독서실·학원 중심의 반복되는 하루. 외부 변수는 적고 자기 규율이 핵심.
- **목표**: 정해진 진도를 매일 빠짐없이, 측정 가능하게 소화하기.
- **페인포인트**: "오늘 8시간 공부했다"는 재지만, 그게 몇 문제·몇 강인지, 어느 과목이 무너졌는지는 흐릿하다. 순공 시간 앱은 시간만 잴 뿐 성취를 못 잰다.
- **대표 정량 목표**: 오전 기출 100문제 / 오후 인강 4강 + 오답 정리 / 저녁 암기 50개.

### 시나리오
> 하루가 거의 고정 템플릿이라 전날 계획은 어제 것을 복제해 미세 조정한다. 대부분 **목표 기준**이다 — "12:00까지 기출 100문제". 오전 프로세스에 진입하자 타이머와 방해금지 모드가 켜지고, 87문제에서 시간이 끝난다. 전환 검증에서 **87%, 원인: '한 지문에 과몰입'**. 오후 인강 4강은 100% 달성인데 예정 2시간을 1시간 40분에 끝내 **20분 단축**. 저녁 마감 대시보드에서 과목별 달성률과 시간 효율이 함께 보인다 — 헌법은 늘 100%에 시간도 단축, 행정법은 3일째 80%대에 시간도 초과. 일기엔 "행정법 페이스가 안 붙는다"고 적는다. 주간 리포트가 행정법 프로세스의 목표량이 과했음을 보여준다.

### 여정 단계
| 단계 | 사용자 행동 | 생각·감정 | 터치포인트 | 페인 & 기회 |
|---|---|---|---|---|
| ① 전날 밤 계획 | 전날 템플릿 복제 후 진도만 조정 | "루틴대로" 담담 | 계획 복제/템플릿 | 반복 하루 → 템플릿·자동 진도 이월 기능 |
| ② 프로세스 진입 | 오전 기출 프로세스 시작 | "100문제 간다" 집중 | 프로세스 카드 | 목표 진도 카운터 필요 |
| ③ 프로세스 실행 | 87문제까지 풂 | "한 문제에 너무 오래…" | 목표 카운터 | 문제 수 증가 입력이 잦음 → 빠른 +1 UX |
| ④ 전환 검증 | 87% + '과몰입' 태그 | "다음 시간에 만회" 담백 | 전환 검증 모달 | 정량 목표라 %가 자동 계산됨(87/100) |
| ⑤ 하루 마감 | 과목별 대시보드 확인 | "행정법이 문제다" 자각 | 대시보드(과목별) | 과목=목표 태그별 집계가 핵심 |
| ⑥ 축적·개선 | 행정법 목표량 하향/시간 증배 | "페이스 맞춰가자" | 주간 과목 리포트 | 과목별 추세선 제공 기회 |

### 핵심 순간
- **정량 목표의 자동 달성률**: 목표가 '100문제'처럼 숫자라 87/100=87%가 자동 계산된다. 이 페르소나에게 가장 자연스럽게 맞는 지점.
- **과목별 대시보드**: 목표 태그(과목)별 집계가 약점 과목을 드러낸다 — 순공 시간 앱이 절대 못 주는 통찰.

## P3 — N잡러/사이드프로젝트 직장인

### 프로필 카드
- **소개**: 본업 개발자이면서 퇴근 후 사이드 프로덕트를 만드는 31세.
- **상황/제약**: 낮은 본업으로 통째로 빠짐. 가용 시간은 새벽·출퇴근·저녁의 파편뿐.
- **목표**: 파편적인 자투리를 모아 사이드 프로젝트를 실제로 전진시키기.
- **페인포인트**: "언젠가 하지"로 미루다 자투리가 증발한다. 파편 시간에 목표를 걸지 않으니 사이드가 늘 제자리.
- **대표 정량 목표**: 새벽(06:00~07:00) 사이드 커밋 1개 / 출근길 관련 강의 1개 / 저녁(21:00~22:30) 기능 1개 완성.

### 시나리오
> 새벽 6시, "07:00까지 커밋 1개"라는 **목표 기준**으로 진입. 타이머·방해금지 모드가 켜지고 30분 만에 커밋 1개, 100% 달성에 **30분 단축**으로 기분 좋게 출근. 출근길 강의 1개 목표는 회의 걱정에 밀려 40%. 전환 검증에서 **40%, 원인: '본업 스트레스로 집중 실패'**. 점심 회의가 취소돼 45분이 빈다 — 이 자투리를 **즉석·시간 기준**으로 "45분에 이슈 2개 정리"로 채운다. 저녁 프로세스는 야근으로 아예 진입 못 함 → 미시작 프로세스로 기록. 마감 대시보드가 "이번 주 저녁 프로세스 달성률 20%"와 "새벽 프로세스는 늘 목표보다 단축"을 나란히 보여준다. 저녁이 구조적으로 불안정함을 깨닫고, 사이드 핵심 작업을 안정적인 새벽으로 옮긴다.

### 여정 단계
| 단계 | 사용자 행동 | 생각·감정 | 터치포인트 | 페인 & 기회 |
|---|---|---|---|---|
| ① 전날 밤 계획 | 새벽·출퇴근·저녁 파편에 목표 배치 | "이번엔 진짜 낸다" 결심 | 계획 편집(자투리 슬롯) | 짧은 슬롯 다수 → 파편 슬롯 템플릿 |
| ② 프로세스 진입 | 새벽 커밋 프로세스 시작 | "30분 안에 1커밋" 몰입 | 프로세스 카드 | 짧은 창이라 진입 지연이 치명적 |
| ③ 프로세스 실행 | 커밋 1개 완료 | "좋아, 됐다" 성취 | 목표 위젯 | 성공 자투리의 즉각 피드백 |
| ④ 전환 검증 | 출근길 40% + '본업 스트레스' | "본업이 잡아먹네" 아쉬움 | 전환 검증 모달 | 미시작 프로세스도 사유 기록 필요 |
| ⑤ 하루 마감 | 시간대별 달성 대시보드 | "저녁이 문제구나" 자각 | 대시보드(시간대별) | 시간대별 안정성 시각화가 핵심 |
| ⑥ 축적·개선 | 핵심 작업을 새벽으로 재배치 | "안정적인 시간에 몰자" | 주간 시간대 리포트 | 시간대 신뢰도 기반 배치 제안 |

### 핵심 순간
- **미시작 프로세스의 처리**: 야근으로 프로세스에 진입조차 못 하는 일이 잦다. 이를 '실패'가 아니라 '구조적 신호'로 기록·집계하는 방식이 이 페르소나 만족을 좌우.
- **시간대 안정성 대시보드**: 어느 시간대가 실제로 믿을 만한지를 보여주면, 사이드 작업 배치 전략이 바뀐다.

## P4 — 프리랜서/1인 창업가

### 프로필 카드
- **소개**: 정해진 출퇴근이 없는 35세 1인 창업가.
- **상황/제약**: 외부에서 부여되는 구조가 전무. 하루의 뼈대를 스스로 세워야 하고, 안 세우면 하루가 녹는다.
- **목표**: 자율성을 유지하되, 딥워크와 잡무·미팅이 뒤섞이지 않게 하루를 규율하기.
- **페인포인트**: 자유롭기에 무너지기도 쉽다. 오전을 흘려보내고 저녁에 후회하는 패턴. 스스로에게 강제할 장치가 없다.
- **대표 정량 목표**: 오전 딥워크(09:00~12:00) 산출물 1건 / 오후 미팅 후 액션아이템 정리 100% / 마케팅 콘텐츠 1개.

### 시나리오
> 외부 마감이 없으니 앱이 유일한 규율 장치다. 오전 딥워크를 **목표 기준**으로 "12:00까지 산출물 1건"으로 걸고 진입하자 타이머와 **방해금지 모드**가 켜진다. 그런데도 메일 알림 유혹에 앱을 나가 이메일에 40분을 뺏긴다(방해금지가 OS 차원 차단까지는 아니라 빠져나갈 틈이 있었다). 결국 산출물은 70%, 타이머 기준 **예정보다 40분 초과**. 전환 검증이 미팅으로 넘어가기 전 막아서고, **70%, 원인: '오전 이메일 침범'** 을 남긴다. 하루 마감 일기에 "오전을 지켜야 한다"고 쓴다. 대시보드는 오전 딥워크가 달성률·시간 효율 모두 이메일 때문에 반복 하락함을 보여준다. 다음날부터 이메일을 오후 프로세스로 격리하고, 방해금지 강도를 'OS 차원 차단'으로 올린다.

### 여정 단계
| 단계 | 사용자 행동 | 생각·감정 | 터치포인트 | 페인 & 기회 |
|---|---|---|---|---|
| ① 전날 밤 계획 | 빈 하루에 딥워크·미팅·잡무 뼈대 세움 | "내일은 오전을 지킨다" 다짐 | 계획 편집(백지) | 외부 구조 없음 → 계획 자체가 최대 가치 |
| ② 프로세스 진입 | 오전 딥워크 시작 | "3시간 몰입" 결의 | 프로세스 카드 | 시작 알림이 유일한 출근 신호 |
| ③ 프로세스 실행 | 이메일에 40분 이탈 | "잠깐만 하려다…" 후회 | 목표 위젯 | 이탈 감지·집중 보호 기회 |
| ④ 전환 검증 | 70% + '이메일 침범' | "또 오전을 놓쳤다" | 전환 검증 모달 | 자기강제 장치로서 가장 중요 |
| ⑤ 하루 마감 | 일기 + 딥워크 추세 대시보드 | "오전 침범이 패턴이다" | 대시보드(프로세스별) | 정성 일기 비중이 큰 페르소나 |
| ⑥ 축적·개선 | 이메일을 오후로 격리 재설계 | "구조를 바꾸자" 결단 | 주간 리포트 | 방해 요인→일과 재설계 제안 |

### 핵심 순간
- **자기강제 장치로서의 전환 검증**: 외부 마감이 없는 이 사용자에게는 전환 관문이 곧 상사이자 마감이다. 제품의 "강제성"이 가장 강하게 요구되는 페르소나.
- **일과 재설계**: 대시보드가 방해 요인의 반복을 드러내면, 단순 회고를 넘어 하루의 구조 자체를 바꾸는 결정으로 이어진다.

## 교차 인사이트 (Cross-Persona Insights)

### 4명을 관통하는 공통 니즈
- **마찰 없는 달성률 입력**: 전환 검증은 모두에게 가치이자 이탈 지점이다. 입력이 몇 초만 무거워도 강제성이 반감한다. → 정량 목표는 자동 %, 원인은 태그 원탭이 기본이어야 한다.
- **실패를 벌이 아닌 데이터로**: 누구도 "미달성"이라는 낙인을 원치 않는다. 달성률·원인을 죄책감이 아니라 **패턴 발견의 재료**로 프레이밍해야 지속된다.
- **마감이 다음 계획의 입력**: 일기·대시보드가 그냥 보고 끝나는 것이 아니라, 다음날 계획에 되먹여질 때 비로소 루프가 닫힌다.
- **시간까지 측정되는 실행**: 타이머와 방해금지 모드가 실행의 기본값이다. "했다/안 했다"가 아니라 "얼마나 빨리·집중해서 했다"가 남아야, 목표를 점점 조여가며 시간 감각을 교정할 수 있다.

### 페르소나별로 갈리는 요구
| 축 | 강함 ← → 약함 |
|---|---|
| 강제성 요구 | P4 프리랜서 · P1 취준생 ↔ P2 수험생(자기규율 이미 강함) |
| 이동/자투리 시간 비중 | P1 · P3 ↔ P2 · P4(고정 장소 중심) |
| 정량 목표의 자연스러움 | P2 수험생(문제 수로 딱 떨어짐) ↔ P4(산출물 '1건'의 질 정의가 모호) |
| 정성 일기 비중 | P4 · P1 ↔ P2(정량 중심) |
| 미시작/구조적 실패 처리 | P3 N잡러(야근으로 통째 결손 잦음)에서 가장 중요 |
| 즉석 계획 비중 | P3 · P1(자투리 잦음) ↔ P2(고정 템플릿) |
| 목표 수립 방향 | 자투리는 시간 기준(P1·P3) ↔ 마감형 목표 기준(P2·P4) |
| 방해금지 강제 수준 | P4(OS 차원 차단 필요) ↔ P2(환경이 이미 통제됨) |

### problem.md 열린 질문과의 연결
- **전환 검증 강제 강도**: P4·P1은 강한 강제가 가치, P2는 이미 규율이 강해 과한 강제가 거슬릴 수 있음 → 강제 강도를 사용자가 조절하는 방향 검토.
- **일기 자동 초안 vs 백지**: P2·P3는 자동 초안이 마찰을 줄여 반가움, P4·P1은 회고 깊이를 위해 백지 선호 가능 → 선택형 검토.
- **대시보드 핵심 지표**: 페르소나마다 결정적 지표가 다름 — P2는 목표(과목)별 달성률, P3는 시간대별 안정성, P1·P4는 실패 원인 분포. → 지표를 페르소나/사용 맥락에 맞게 구성 가능하게.
- **미시작 프로세스 처리**: P3에서 부상 — '진입조차 못 함'을 실패가 아닌 구조적 신호로 다루는 모델 필요.
- **방해금지 강제 수준**: P4는 OS 차원 앱 차단이 필요할 만큼 절실, P2는 이미 환경이 통제돼 앱 내 표시로 충분 → 강도 조절 옵션.
- **타이머 종료 누락 보정**: 이동·자투리가 많은 P1·P3는 타이머 끄는 걸 잊기 쉬움 → 자동 종료·소요 시간 보정 로직 필요.
- **즉석 vs 사전 구분 UX**: 즉석 비중이 큰 P3에게는 '몇 초 안에 목표 수립'과 두 방향(시간/목표 기준) 빠른 선택이 핵심.
