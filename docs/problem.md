# 문제 정의 (Problem Statement)

## 한 줄 요약

일정관리 앱은 넘쳐나지만, **하루를 프로세스 단위로 쪼개고 각 프로세스마다 정량적 목표를 걸어 "극한의 계획 → 실행 → 검증"을 강제하고, 하루가 끝나면 그 기록을 일기와 대시보드로 마감해 다음날 계획에 되먹이는 앱은 없다.**

---

## 1. 배경: 내가 하루를 계획하는 방식

- 전날 밤, 다음날의 하루 흐름을 설계한다. 대략적인 할 일 목록이 아니라, 하루 전체를 프로세스로 짠다. 다만 **모든 시간을 전날에 확정할 필요는 없다** — 어떤 구간은 일부러 비워 두고, 당일 그 시간이 다가왔을 때 목표를 수립한다. (아래 '계획 시점' 참고)
- 하루는 연속된 **프로세스(process)의 흐름**이다.
  - 예: `기상 → 이동 → 출근(업무) → 점심 → 업무 → 퇴근 이동 → 저녁 → 취침 전`
- 각 프로세스는 시간 구간을 가진다. 이동거리·이동시간까지 계산해서 구간을 확정한다.
  - 예: 8시~9시는 회사로 가는 이동 시간.
- 그리고 핵심: **각 프로세스마다 "그 시간 안에 해내야 하는 정량적 목표"를 미리 정한다.**

### 구체적 예시

> **이동 프로세스 (08:00 ~ 09:00)**
> - 실리콘밸리 개발자 영상 **2개** 보기
> - 지금 읽는 책 **200페이지까지** 읽기
>
> 목표가 "영상 좀 보기", "책 읽기"가 아니다. **개수, 페이지, 횟수** 등 달성 여부를 판정할 수 있는 수치다.

### 계획 시점: 사전 계획 vs 즉석 계획

- 하루의 뼈대와 확실한 프로세스는 **사전 계획(pre-planned)** — 전날 밤에 목표까지 확정한다.
- 그러나 유동적이거나, 그때의 컨디션·상황을 봐야 목표를 정할 수 있는 구간은 **즉석 계획(just-in-time)** — 당일 그 시간이 다가오면 그 자리에서 목표를 수립하고 바로 실행한다.
- 즉, 모든 목표를 전날에 완성할 필요는 없다. 시간 구간(프로세스)만 잡아 두고 **목표는 진입 시점에 채우는 방식**이 허용된다.

### 목표 수립의 두 방향: 시간 기준 vs 목표 기준

목표를 세울 때 방향이 두 가지다. 특히 즉석 계획에서 유용하다.

- **시간 기준 (Time-box)**: *"이번에 나에게 주어진 시간이 이만큼인데, 어디까지 해낼 수 있지?"* — 주어진 시간을 먼저 보고, 그 안에서 달성할 정량 목표를 정한다.
  - 예: 지금 자투리 40분이 생겼다 → "이 40분에 CS 문제 10개".
- **목표 기준 (Target-box)**: *"나는 여기까지는 반드시 해내고야 만다."* — 반드시 끝낼 정량 목표를 먼저 정하고, 마감 시각을 붙인다.
  - 예: "14:30까지 A회사 자기소개서 다 쓰기".

### 실행 타이머와 시간 효율

목표를 시작하는 순간부터 **타이머가 돌고, 방해금지(집중) 모드가 자동으로 켜진다.** 목표를 끝내면 타이머가 멈춘다.

- 실제 소요 시간 = (종료 시각 − 시작 시각).
- 이걸 계획/목표 시간과 비교한다:
  - 목표보다 **빨리** 끝냈으면 → 얼마나 **단축**했는지 (예: 60분 목표 → 45분, 15분 단축)
  - 목표보다 **늦게** 끝났으면 → 얼마나 **초과**했는지
- 이 시간 효율도 하루 마감 대시보드에서 확인한다.

### 프로세스 전환 시 리뷰

다음 프로세스로 넘어갈 때, 이전 프로세스를 그냥 지나치지 않는다:

1. 목표를 이뤘는가? (달성 / 미달성)
2. 미달성이라면 **몇 %를 달성했는가?**
3. **무엇이 문제였는가?** (방해 요인, 잘못된 목표 설정, 컨디션 등)
4. 정리가 끝나야 다음 프로세스로 진행한다.

즉, 하루가 `계획 → 실행 → 측정 → 회고`의 마이크로 사이클이 프로세스 개수만큼 반복되는 구조다.

### 하루의 마감

하루는 마지막 프로세스가 끝난다고 그냥 끝나지 않는다. 하루 전체를 닫는 **마감 루프**가 있다:

1. **오늘의 일기**를 쓴다. (정성적 기록 — 오늘 하루가 어땠는지, 무엇을 느꼈는지)
2. 오늘 세운 목표 대비 실제 달성 내역을 **대시보드**로 한눈에 본다. (정량 집계 — 프로세스별 달성률, 하루 총 달성률, 실패 원인 분포 등)

이 대시보드의 데이터는 하루 종일 프로세스 전환 때마다 입력한 달성률·실패 원인이 그대로 소스가 된다. 별도로 다시 기록하는 것이 아니라, **낮 동안 쌓인 검증 데이터가 저녁에 자동으로 집계된다.**

정리하면 이 시스템은 2층 사이클이다:

- **미시 사이클** (프로세스마다): `계획 → 실행 → 측정 → 회고`
- **거시 사이클** (하루마다): `하루 종료 → 일기 → 대시보드 → 다음날 계획에 반영`

## 2. 문제: 기존 일정관리 앱은 이 방식을 지원하지 못한다

| 기존 앱의 모델 | 내가 필요한 모델 |
|---|---|
| 캘린더에 이벤트를 "배치"한다 | 하루를 끊김 없는 **프로세스 체인**으로 설계한다 |
| 할 일은 체크박스 (했다/안 했다) | 목표는 **정량 수치** (영상 2개, 200pg) — 부분 달성률(%)이 존재한다 |
| 시간이 지나면 그냥 다음 일정 | 프로세스 전환 시 **달성률 입력과 실패 원인 정리를 강제** |
| 이동 시간은 빈 공백 | 이동 시간도 목표가 걸리는 **하나의 프로세스** |
| 회고는 별도 앱(일기, 저널)의 몫 | 회고가 하루 흐름 **안에 내장**되어 전환의 조건이 됨 |
| 일기·성취 분석은 각각 다른 앱에서 수동 정리 | 하루 흐름 데이터가 그대로 일기·대시보드로 **자동 집계**되어 마감됨 |
| 일정은 미리 다 짜야 함 (고정) | 뼈대만 잡고 **목표는 당일 즉석 수립** 가능 (사전+즉석 혼합) |
| 타이머는 "몇 분 썼나"만 잼 | 목표 시간 대비 **단축/초과까지 측정**해 시간 효율을 남김 (실행 중 방해금지 모드) |

기존 앱들의 근본적 한계:

- **투두 앱** (Todoist, Things 등): 체크박스는 이진적이다. "70% 달성, 원인은 지하철에서 졸음"을 표현할 수 없다.
- **캘린더 앱** (Google Calendar 등): 시간 블록은 만들 수 있지만, 블록에 정량 목표를 걸고 종료 시 검증하는 개념이 없다.
- **타임블로킹/포모도로 앱**: 시간을 쓰는 것 자체에 집중할 뿐, "그 시간에 얼마만큼 해냈는가"를 측정하지 않는다.
- **습관 트래커**: 반복 습관에는 맞지만, 매일 달라지는 하루 흐름 설계와 프로세스별 목표에는 맞지 않는다.

결과적으로, 나 같은 사람은 캘린더 + 투두 + 메모 + 일기를 조합해서 수동으로 이 시스템을 돌리고 있고, **전환 시 검증을 강제하는 장치가 없어서 시스템이 쉽게 무너진다.**

## 3. 핵심 개념 정리

이 앱이 다루는 세계는 다섯 가지 개념으로 요약된다.

### 프로세스 (Process)
- 하루를 구성하는 시간 구간 단위. (기상, 이동, 업무, ...)
- 시작/종료 시각을 가지며, 이동거리·이동시간 같은 현실 제약을 반영해 설계된다.
- 하루는 프로세스들의 **순서 있는 체인**이다. 공백 없이 이어진다.
- 계획 시점은 두 가지다: **사전 계획**(전날 목표까지 확정) / **즉석 계획**(당일 진입 시점에 목표 수립).

### 정량 목표 (Quantitative Goal)
- 각 프로세스에 걸리는, **수치로 달성 여부를 판정할 수 있는** 목표.
- "책 읽기" ❌ → "책 200pg까지 읽기" ✅
- 하나의 프로세스에 여러 개의 목표가 걸릴 수 있다.
- 수립 방향이 두 가지다:
  - **시간 기준 (Time-box)**: 주어진 시간을 보고 "이 안에 어디까지" 목표를 정함.
  - **목표 기준 (Target-box)**: "여기까지 반드시" 목표를 먼저 정하고 마감 시각을 붙임.

### 전환 검증 (Transition Review)
- 프로세스가 끝나고 다음 프로세스로 넘어가는 순간에 발생하는 **강제 관문**.
- 입력 항목:
  - 목표별 달성 여부 / 달성률(%)
  - 미달성 시 원인 정리
- 이 관문을 통과해야 다음 프로세스가 시작된다. 회고를 "나중에 몰아서" 하는 것이 아니라 **전환의 조건**으로 만든다.

### 실행 타이머 & 집중 모드 (Focus Timer)
- 목표를 시작하면 **타이머가 시작되고 방해금지 모드가 자동으로 켜진다.** 종료하면 타이머가 멈춘다.
- 이로써 **계획 시간 대비 실제 소요 시간**을 측정한다 — 단축했는가, 초과했는가.
- 두 목표 방향과 맞물린다: 시간 기준(Time-box)이면 "정해진 시간 안에 목표를 얼마나 채웠나", 목표 기준(Target-box)이면 "정한 목표를 예정보다 빨리/늦게 끝냈나"를 잰다.
- 이 **시간 효율** 데이터가 대시보드에서 달성률과 나란히 놓이는 또 다른 축이 된다.

### 하루 마감 (Daily Close: 일기 + 대시보드)
- 하루가 끝날 때 하루 전체를 닫는 마감 단위. 두 가지 형태의 기록으로 구성된다.
- **일기 (정성)**: 오늘 하루의 서사와 감정, 맥락을 남긴다. 숫자로 잡히지 않는 부분을 붙든다.
- **대시보드 (정량)**: 낮 동안 전환 검증에서 쌓인 데이터를 자동 집계해 보여준다.
  - 프로세스별 달성률, 하루 총 달성률
  - **시간 효율**: 목표 시간 대비 단축/초과 (얼마나 빨리·느리게 해냈는가)
  - 시간대별 집중도 / 목표 달성 패턴
  - 실패 원인 태그 분포 (무엇이 반복적으로 나를 막았는가)
- 핵심: 대시보드는 **새로 입력하는 화면이 아니라, 전환 검증 데이터가 소스인 집계 화면**이다. 그리고 이 마감 결과가 다음날 계획의 입력이 된다.

## 4. 왜 "극한(extreme)"인가

- 하루의 **모든 시간**이 계획 대상이다. 이동 시간, 자투리 시간도 예외가 아니다.
- 모든 목표가 **측정 가능**해야 한다. 애매한 목표는 이 시스템에 들어올 수 없다.
- 실패를 허용하되, **실패를 그냥 넘기는 것은 허용하지 않는다.** 몇 %였는지, 왜였는지를 반드시 남긴다.
- 타겟 사용자는 "일정을 관리하고 싶은 사람"이 아니라, **하루를 설계하고 그 설계를 검증하며 사는 사람**이다.

## 5. 이 문제를 풀었을 때의 가치 (가설)

- **계획의 질 향상**: 달성률 데이터가 쌓이면 "나는 이동 1시간에 영상 2개는 무리, 1개가 적정"처럼 목표 설정 자체가 현실화된다.
- **실행력 강화**: 전환 관문이 있다는 사실 자체가 프로세스 중의 집중을 만든다 (측정된다는 것을 알면 행동이 달라진다).
- **실패 패턴의 가시화**: 실패 원인이 프로세스·시간대별로 축적되어, 반복되는 방해 요인을 구조적으로 제거할 수 있다.
- **회고의 습관화 + 성취의 가시화**: 일기로 하루의 정성적 맥락을 보존하고, 대시보드로 하루의 성취를 눈으로 확인한다. "오늘 이만큼 해냈다"가 매일 보이면 지속 동기가 생기고, 마감이 다음날 계획의 출발점이 되어 루프가 닫힌다.
- **시간 감각의 교정**: 목표 시간 대비 실제 소요를 매번 재면, "일이 주어진 시간을 늘 꽉 채우는"(파킨슨 법칙) 관성을 깨고 마감을 걸어 스스로를 압축하게 된다. 시간을 단축한 경험이 데이터로 쌓이는 것 자체가 보상이 된다. 방해금지 모드는 이 측정이 왜곡되지 않도록 실행 중 집중을 지킨다.

## 7. 열린 질문 (Open Questions)

- 전환 검증을 얼마나 "강제"할 것인가? (스킵 불가 vs 페널티 vs 리마인드)
- 프로세스가 계획보다 일찍/늦게 끝나는 경우 체인을 어떻게 재조정하는가?
- 정량 목표의 달성률 입력을 최대한 마찰 없이 만드는 UX는 무엇인가?
- 이동시간 계산을 지도 API 연동으로 자동화할 것인가, 수동 입력으로 시작할 것인가?
